Ventaja clave

El LLM no necesita "saber" el contenido de antemano.
Los documentos se actualizan sin reentrenar el modelo.

En este sistema

Toda la inferencia es externa al contenedor.
El servidor LM Studio está en openai.ull.es:8080.

def main(): configurar_modelos() listado = leer_archivos_carpeta(...) if listado: # hay ficheros nuevos nodes = digitaliza_documentos(...) if nodes: if _indice_existe(): index = actualizar_indice( cargar_indice(), nodes) else: index = construir_indice(nodes) elif _indice_existe(): index = cargar_indice() else: return # nada que hacer else: # sin ficheros nuevos if not _indice_existe(): return index = cargar_indice() mostrar_reporte_indice(index) chatear(index, preguntas)

Los 5 escenarios

  • A Docs nuevos + sin índice → construir
  • B Docs nuevos + índice previo → cargar + actualizar
  • C Docs sin nodos + índice previo → cargar
  • D Sin docs + índice previo → cargar
  • E Sin docs + sin índice → salir
El script es idempotente:
si no hay trabajo nuevo simplemente carga el índice y responde.

Formatos de documentos soportados

Patrón dispatcher — añadir un nuevo formato solo requiere una función y un elif

📄 PDF

Librería: pypdf.PdfReader
Extrae texto página a página.
En modo semántico: concatena todo el documento antes de dividir.

📝 DokuWiki

Extensión: .dokuwiki
Convierte con pypandoc a texto plano.
Formato nativo de la Wiki del STIC.

📋 Markdown

Extensión: .md
Convierte con pypandoc a texto plano.
Mismo núcleo que DokuWiki.
def digitaliza_un_documento(documento, fila_indice): ext = documento["extension"].lower() if ext == "pdf": return _digitaliza_pdf(documento, fila_indice) elif ext == "dokuwiki": return _digitaliza_dokuwiki(documento, fila_indice) elif ext == "md": return _digitaliza_markdown(documento, fila_indice) else: print(f" Extensión no soportada: '{ext}'"); return []
Los documentos no registrados en indice.xlsx se omiten y permanecen en Nuevos documentos/.
Los procesados se mueven automáticamente a Documentos/.

Troceado fijo vs semántico

Modo fijo

MODO_TROCEADO = "fijo"
  • Ventana de 3000 palabras con solapamiento de 50
  • Divide por página en PDFs
  • Tamaño predecible y uniforme
  • Puede cortar a mitad de frase

Modo semántico

MODO_TROCEADO = "semantico"
  • SemanticSplitterNodeParser de LlamaIndex
  • Usa el modelo de embeddings para detectar cambios de tema
  • Trozos más coherentes semánticamente
  • Más lento (llama al servidor para cada documento)
Resultado modo fijo — mismos documentos
Nodos generados: 69 Trozos de tamaño uniforme ~300 palabras 14 trozos "cortos" de basura tipográfica: ñ é Ó • • • •
Resultado modo semántico
Nodos generados: 37 Trozos de tamaño variable (34–1193 palabras) Solo 3 trozos cortos eliminados → secciones temáticas completas

Filtrado de trozos cortos

El problema

Algunos PDFs mal formateados generan trozos con ruido tipográfico:

[TROZO CORTO] 2025-01-3 (2 palabras): 'ñ é' [TROZO CORTO] 2025-01-9 (1 palabras): 'ñ' [TROZO CORTO] 2025-01-4 (8 palabras): '• • • • • • • •'

Indexar esto contamina el espacio vectorial con ruido.

La solución

UMBRAL_VISUALIZACION_PALABRAS = 30 def _filtrar_trozos_cortos(resultado): for i, item in enumerate(resultado): n = len(item["texto"].split()) es_ultimo = (i == len(resultado) - 1) if n < UMBRAL and not es_ultimo: print("[TROZO CORTO ELIMINADO]", ...) continue # descartado filtrados.append(item)
Regla de oro: el último trozo siempre se conserva — es el cierre natural del documento, aunque sea breve.

Embeddings e índice vectorial

¿Qué es un embedding?

Una representación numérica de texto: frases con significado similar quedan cerca en el espacio vectorial.

"vacaciones PTGAS 2026"
↓ texto → vector de N dimensiones
[0.12, -0.34, 0.89, ...]

Clase LMStudioEmbedding

  • Hereda de BaseEmbedding de LlamaIndex
  • Llama a /v1/embeddings vía HTTP REST
  • Token de autorización + timeout de 600 s

Persistencia del índice

El índice se guarda en disco para no recalcular embeddings en cada ejecución

  • docstore.json — nodos con texto y metadatos
  • index_store.json — estructura del índice vectorial
  • vector_store.json — vectores (embeddings)
  • graph_store.json — relaciones entre nodos

Retriever — filtros y control de calidad

Tres capas de filtrado

1. MetadataFilter — Estado = "Vigente"
Excluye documentos obsoletos antes de buscar
↓
2. similarity_top_k = 4
El retriever devuelve como máximo 4 nodos
↓
3. SimilarityPostprocessor (≥ 0.40)
Descarta nodos poco relevantes → 0–4 nodos finales
MetadataFilters(filters=[ ExactMatchFilter(key="Estado", value=FILTRO_ESTADO) ]) index.as_chat_engine( chat_mode="context", similarity_top_k=RETRIEVER_TOP_K, filters=filtros, node_postprocessors=[ SimilarityPostprocessor( similarity_cutoff=RETRIEVER_SCORE_MINIMO) ], system_prompt="...", )
Score = similitud coseno entre el embedding de la pregunta y el del trozo.
0 = sin relación  ·  1 = idénticos
