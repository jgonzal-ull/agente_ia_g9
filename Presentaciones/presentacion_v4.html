¿Qué hay de nuevo en la v4.0?

Toda la base de la v3.0 se mantiene intacta (ingesta, troceado, persistencia, retriever). La v4.0 añade cinco bloques nuevos.

🤖 Front-end Telegram

  • iniciar_bot()
  • comando_start()
  • manejar_mensaje()
  • error_handler()

🔗 Puente bot → RAG

  • obtener_respuesta_ia()
  • chatear_simple()
  • sustituye al chatear() de consola de la v3.0

🔒 Seguridad

  • _parse_lista_blanca()
  • lista_blanca
  • TELEGRAM_TOKEN y TELEGRAM_WHITELIST en .env

🔄 Coherencia de datos

  • sincronizar_estado_con_indice() — alinea el metadato Estado de los chunks con indice.xlsx
  • _acceder_nodos_docstore()

🛠️ Infraestructura

  • logging + logger — trazas con timestamp
  • index pasa a ser variable global
  • main() arranca el bot al final

El cambio clave: de consola a Telegram

En la v3.0 el agente respondía a una lista fija de preguntas por consola. En la v4.0 atiende a usuarios reales por chat.

v3.0 — consola

# preguntas fijas en el código chatear(index, [ "¿Quién es el responsable…?", "¿Cómo es la estructura…?", ])
  • Un solo usuario (quien ejecuta)
  • Preguntas escritas en main()
  • Salida por print()

v4.0 — bot de Telegram

# servidor que escucha mensajes iniciar_bot() # → application.run_polling() # atiende a cualquier usuario # autorizado, 24/7
  • Múltiples usuarios autorizados
  • Preguntas libres por chat
  • Respuesta enviada a Telegram
El pipeline RAG no cambia: el bot es solo una nueva puerta de entrada a las mismas funciones de recuperación y generación.

Paso previo: dar de alta el bot

El token del bot lo entrega BotFather, el bot oficial de Telegram para gestionar bots.

1. Hablar con @BotFather → comando /newbot
↓
2. Elegir un nombre (p. ej. «Test Agente IA G9»)
↓
3. Elegir un username terminado en bot
↓
4. BotFather devuelve el token HTTP API

El token tiene este formato

Use this token to access the HTTP API: 123456789:AAH-EjemploDeTokenNoReal_xyz Keep your token secure and store it safely.
⚠️ El token da control total del bot.
Nunca se escribe en el código ni se sube a git: va en .env.

El nuevo main()

Mantiene el flujo diario de la v3.0 y añade 2 pasos nuevos al final.

def main(): global index # ← NUEVO configurar_modelos() # 1-2. Ingesta + construir/cargar índice # (idéntico a la v3.0) listado = leer_archivos_carpeta(...) if listado: nodes = digitaliza_documentos(...) ... index = construir_indice(nodes) else: index = cargar_indice() # 3. Coherencia de Estado ← NUEVO sincronizar_estado_con_indice(index) # 4. Reporte (igual que v3.0) mostrar_reporte_indice(index) # 5. Arrancar el bot ← NUEVO iniciar_bot()

Qué cambia respecto a la v3.0

  • + global index — el índice deja de ser local para que lo vean los handlers del bot
  • + sincronizar_estado_con_indice(index) antes del reporte
  • − desaparece chatear(index, [preguntas])
  • + en su lugar, iniciar_bot()
Idea: primero se prepara el conocimiento (ingesta + índice + sincronización) y, una vez listo, se levanta el bot para servir consultas.

iniciar_bot() — arranque y registro de handlers

def iniciar_bot(): if not TELEGRAM_TOKEN: print("TELEGRAM_TOKEN no definido…"); return application = Application.builder().token(TELEGRAM_TOKEN).build() application.add_handler(CommandHandler("start", comando_start)) # /start application.add_handler(MessageHandler(filters.TEXT & ~filters.COMMAND, manejar_mensaje)) # texto normal application.add_handler(MessageHandler(filters.COMMAND, error_handler)) # comandos sueltos application.add_error_handler(error_handler) # resto de errores application.run_polling() # bucle que sondea Telegram en busca de mensajes

Cada handler atiende un tipo de evento

  • CommandHandler("start", …) → el comando /start
  • filters.TEXT & ~filters.COMMAND → mensajes de texto que no son comandos
  • run_polling() es bloqueante: el programa se queda aquí escuchando
filters.TEXT & ~filters.COMMAND
filtro de texto Y (&) negación (~) de comando

Así /start lo atiende su CommandHandler y el resto del texto va a manejar_mensaje.

Handlers async: comando_start y error_handler

Los manejadores de python-telegram-bot son corrutinas (async def) con la firma (update, context).

Bienvenida al /start

async def comando_start(update, context): user = update.effective_user await update.message.reply_html( rf"¡Hola {user.mention_html()}! " "Soy tu bot IA. Envíame un " "mensaje y te responderé.")
  • update = el mensaje/evento entrante
  • reply_html permite formato (negrita, enlaces)
  • mention_html() menciona al usuario por su nombre

Registro de errores

async def error_handler(update, context): logger.error( f"Update {update} causó " f"error {context.error}")
Captura cualquier excepción no controlada durante el procesado de un update y la deja en el log, en lugar de tumbar el bot.
await se usa porque enviar a Telegram es una operación de red asíncrona.

manejar_mensaje() — el corazón del bot

Se ejecuta con cada mensaje de texto. Comprueba la lista blanca antes de responder.

async def manejar_mensaje(update, context): mensaje_recibido = update.message.text chat_id = update.message.chat_id if chat_id in lista_blanca: # ✓ usuario autorizado logger.info(f"Mensaje recibido de {chat_id}: '{mensaje_recibido}'") respuesta_ia = obtener_respuesta_ia(mensaje_recibido) # → RAG else: # ✗ no autorizado respuesta_ia = (f"{update.effective_user.first_name} con id " f"{update.effective_user.id} no estás autorizado para el uso " "de este agente. Ponte en contacto con el administrador…") await update.message.reply_text(respuesta_ia) # responde en el chat

la pregunta va al agente RAG
✗ No autorizado
mensaje de rechazo con su id
Siempre
reply_text contesta en el chat

La lista blanca — _parse_lista_blanca()

Los chat_id autorizados se leen del entorno, no del código, para no publicar datos personales.

def _parse_lista_blanca(raw): """'id:nombre,id:nombre' → {int(id): nombre}""" autorizados = {} for entrada in raw.split(","): entrada = entrada.strip() if not entrada: continue id_str, _, nombre = entrada.partition(":") try: autorizados[int(id_str)] = nombre.strip() except ValueError: print("… id no numérico, ignorado") return autorizados

Formato en .env

TELEGRAM_WHITELIST=111:Ana,222:Luis
se convierte en…
{ 111: "Ana", 222: "Luis" }

obtener_respuesta_ia() — el puente al RAG

Decide si responde con un atajo rápido o consulta de verdad al motor RAG.

def obtener_respuesta_ia(mensaje_usuario): logger.info(f"Mensaje para la IA: …") if "hola" in mensaje_usuario.lower(): respuesta = "¡Hola! Soy tu asistente…" elif "cómo estás" in …: respuesta = "Estoy funcionando…" elif "adiós" in …: respuesta = "¡Hasta pronto!" else: respuesta = chatear_simple( index, Settings.llm, mensaje_usuario) return respuesta

Dos caminos

  • Atajo saludos como «hola», «cómo estás», «adiós» → respuesta fija e instantánea, sin gastar el LLM
  • RAG cualquier otra cosa → chatear_simple() con el índice global y Settings.llm
Aquí es donde se usa la variable global index: el handler no recibe el índice por parámetro, lo toma del ámbito global preparado en main().

chatear_simple() — consulta y referencias

Es el chatear() de la v3.0 adaptado al bot: en vez de imprimir, devuelve un texto listo para enviar.

chat_engine = indice.as_chat_engine( chat_mode="context", similarity_top_k=RETRIEVER_TOP_K, filters=filtros, # Estado=Vigente node_postprocessors=[…0.40…], system_prompt="…cita tus fuentes…") obj_response = chat_engine.chat(pregunta) responses.append(str(obj_response.response)) for sn in obj_response.source_nodes: url = sn.node.metadata.get("URL") referencias.append(f"- {url}") return "\n".join(responses + referencias)

Mismos filtros que la v3.0

  • Estado = "Vigente" · top_k = 4 · score ≥ 0.40

Dos diferencias importantes

  • ↩ devuelve un str (no imprime) — porque reply_text necesita una cadena
  • 🔗 añade un bloque Referencias con las URL de las fuentes
El detalle verboso (score, metadatos, texto) va al log, no al chat: Telegram corta los mensajes a 4096 caracteres.

Sincronización de Estado — el problema

Un documento puede dejar de estar vigente. Se cambia su Estado en indice.xlsx…

indice.xlsx Identificador Estado 2025-06 No vigente ← cambiado

…pero los chunks ya indexados conservan el valor antiguo en su metadato:

data_storage_lmstudio/ (chunks de 2025-06) 2025-06-0 Estado = Vigente ← desactualizado 2025-06-1 Estado = Vigente ← desactualizado

La consecuencia

El retriever filtra por Estado = "Vigente".

Si los chunks no se actualizan, el agente seguiría citando un documento ya obsoleto.

La meta

Que el metadato Estado de todos los chunks sea siempre coherente con indice.xlsx, en ambos sentidos (Vigente ⇄ No vigente).

sincronizar_estado_con_indice() — la solución

# 1. Mapa Identificador → Estado (del xlsx) df = pd.read_excel(FICHERO_INDICE, …) mapa_estado = { str(id): str(estado) … } # 2. Recorrer TODOS los chunks del store for node in _acceder_nodos_docstore(docstore): ident = node.metadata.get("Identificador") correcto = mapa_estado.get(ident) actual = node.metadata.get("Estado") if actual != correcto: # 3. corregir node.metadata["Estado"] = correcto nodes_a_actualizar.append(node) # 4. Persistir SOLO si hubo cambios if nodes_a_actualizar: docstore.add_documents(nodes_a_actualizar) index.storage_context.persist(…)

Cuatro pasos

  • 1 Lee indice.xlsx → diccionario Identificador → Estado
  • 2 Recorre los nodos del store (_acceder_nodos_docstore tolera docs / _docs)
  • 3 Compara y corrige solo los que difieren
  • 4 Re-persiste únicamente si hubo cambios
Es genérico: funciona para cualquier Identificador y en ambos sentidos. Avisa de chunks sin Identificador o ausentes del índice.
[2025-06] 2025-06-0: 'Vigente' → 'No vigente' Metadatos 'Estado' sincronizados y store persistido.

Logging estructurado

La v4.0 sustituye muchos print por logger: trazas con marca de tiempo, imprescindibles para depurar un bot que corre desatendido.

Configuración (una vez)

logging.basicConfig( format="%(asctime)s - %(name)s - " "%(levelname)s - %(message)s", level=logging.INFO) logger = logging.getLogger(__name__)

Uso en el código

logger.info(f"Mensaje recibido de {chat_id}") logger.info("Creando el Chat Engine…") logger.error(f"… causó error {context.error}")
